# Dashboard — Cyber Agent Engine

Web interface for the multi-agent coordinator.

## Architecture

```
dashboard/
├── app.py          ← FastAPI API (system status, coordinator proxy, SSE)
├── templates/      ← old static index.html (replaced by frontend/dist)
├── static/         ← Svelte build generated by `npm run build` (served by FastAPI)
└── frontend/       ← Svelte application (sources)
    ├── src/
    │   ├── App.svelte                       ← tab-based routing, SSE connection on mount
    │   ├── lib/
    │   │   ├── components/
    │   │   │   ├── Header.svelte
    │   │   │   ├── Sidebar.svelte           ← nav: Chat / Tasks / Logs / Status / About
    │   │   │   ├── ChatView.svelte          ← coordinator command interface
    │   │   │   ├── AgentStatusView.svelte   ← real-time agent tasks (from SSE)
    │   │   │   ├── SystemStatusView.svelte  ← CPU / RAM / network (5s polling)
    │   │   │   └── NotificationDisplay.svelte
    │   │   ├── stores/
    │   │   │   ├── notificationStore.ts     ← notify(msg, type), auto-clear 5s
    │   │   │   ├── taskStore.ts             ← Task[], upsertTask()
    │   │   │   └── sidebarStore.ts
    │   │   └── utils/
    │   │       └── coordinatorApi.ts        ← connectSSE() + sendCommand()
    ├── package.json
    └── vite.config.ts                       ← proxy /api and /events → coordinator:3001
```

## Frontend Stack

- **Svelte 4** + TypeScript
- **Tailwind CSS 3** (dark mode `class`, zinc theme)
- **Vite 4** (build + dev server)
- No Tauri — pure web SPA served by FastAPI

## Testing with the Interface

The full stack requires **3 consoles**.

```
Console 1 — Tool-agent server (OPNsense / WireGuard / CrowdSec) — UDS socket
Console 2 — Coordinator — port 3001 (vLLM loaded internally on startup)
Console 3 — Dashboard FastAPI — port 8080
```

> **Frontend dev mode**: an optional 4th console runs the Vite server (`:5173`)
> with hot-reload instead of serving the static build from FastAPI.

The LLM backend (`COORDINATOR_BACKEND=vllm`) is instantiated **in-process** by the coordinator via
`NativeVLLMClient` — no external LLM service is required. The model loads on coordinator startup
(a few seconds depending on available VRAM).

To use another backend, see the LLM Backends section.

---

### Console 1 — Tool-agent server (UDS)

`server.py` is a **monolithic server**: it hosts all 3 agents (OPNsense, WireGuard, CrowdSec)
in the same process, on a single Unix socket. The coordinator believes it has 3 separate servers,
but `WIREGUARD_AGENT_SOCK` and `CROWDSEC_AGENT_SOCK` both point to the same socket as
`OPNSENSE_AGENT_SOCK` — routing to the correct agent is handled internally by the classification in `server.py`.

The path is defined by `UDS_SOCKET_PATH` in `.env` (currently `/tmp/cyber-agents/opnsense.sock`).

```bash
cd /srv/cyber-agent-engine
source venv/bin/activate
source .env
python server.py
# → "Tool-agent server starting on UDS: /tmp/cyber-agents/opnsense.sock"
```

Verification (curl with Unix socket):
```bash
curl --unix-socket /tmp/cyber-agents/opnsense.sock http://localhost/capabilities
```

> If `UDS_SOCKET_PATH` is not defined, the server falls back to TCP on the `TOOL_AGENT_PORT` (default: 3000) — dev fallback only.

### Console 2 — Coordinator (port 3001)

```bash
cd /srv/cyber-agent-engine
source venv/bin/activate
source .env
uvicorn coordinator.server:app --host 0.0.0.0 --port 3001
```

On startup, the coordinator loads the vLLM model (default `Qwen/Qwen2.5-7B-Instruct`,
configurable via `COORDINATOR_MODEL`). VRAM consumption depends on `COORDINATOR_GPU_UTIL` (default: 0.5)
and `VLLM_MAX_MODEL_LEN` (default: 4096 in the current `.env`).

Verification: `curl http://localhost:3001/coordinator/health`

### Recommended Startup Order

```
Tool-agent → Coordinator → Dashboard
```

The coordinator attempts to connect to agents on startup (lifespan).
If an agent is missing, it is ignored with a warning — the coordinator will still start.

## Expected Coordinator API

The frontend expects two routes on the coordinator side:

| Route | Method | Description |
|---|---|---|
| `/events` | `GET` (SSE) | Stream of `task_update` and `notification` events |
| `/api/command` | `POST` JSON `{ command }` | Sends a command, returns `{ reply }` |
| `/api/status` | `GET` | System stats (CPU, RAM, network, vLLM) |

### SSE Format

```json
{ "type": "task_update", "task": { "id": "...", "agent": "opnsense", "description": "...", "status": "running", "created_at": "..." } }
{ "type": "notification", "message": "...", "status": "success" }
```

## Audit — Known Bugs and Points of Attention

### ~~Critical bug: checkpoint routes not relayed by `app.py`~~ — fixed

`app.py` now exposes `POST /coordinator/checkpoint/{run_id}/approve` and `/reject`
which delegate to the coordinator on the same model as `/api/command`.

### ~~SSE without heartbeat — risk of suspended stream~~ — fixed

The SSE relay now uses `httpx.Timeout(read=20.0)` + automatic reconnection loop.
In case of coordinator silence → keepalive `": keepalive\n\n"` is sent to the client.
In case of exception → SSE error message + retry after 3s.

### ~~SSE notifications in a loop~~ — fixed

`coordinatorApi.ts` uses the `sseStore.ts` store (`SseState : connecting | connected | disconnected`).
Error toast displayed only on the **first** disconnection; success toast on reconnection.
Silent retries from `EventSource` no longer trigger notifications.

### ~~`scrollToBottom` with `setTimeout(50ms)`~~ — fixed

`ChatView.svelte` now uses `await tick()` (imported from `svelte`).

### Authentication — Decision and Roadmap

#### Current choice: no auth (single-user LAN usage)

Acceptable as long as the dashboard is only accessible on the local network.
If the dashboard needs to be exposed, implement Option A below.

---

#### ~~Option A — Static Bearer token~~ — implemented

Mechanism: token in `.env`, verified by a FastAPI dependency on all protected routes.
`EventSource` does not support custom headers → token is passed as a query param for `/events`.

**`app.py`** — add:

```python
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

_bearer = HTTPBearer(auto_error=False)
_DASHBOARD_TOKEN = os.getenv("DASHBOARD_TOKEN")

def require_auth(
    creds: HTTPAuthorizationCredentials | None = Depends(_bearer),
    token: str | None = Query(default=None),   # fallback query param for SSE
) -> None:
    if not _DASHBOARD_TOKEN:
        return  # auth disabled if variable is not set (dev mode)
    provided = (creds.credentials if creds else None) or token
    if provided != _DASHBOARD_TOKEN:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Invalid token")
```

Apply the dependency:

```python
@app.post("/api/command", dependencies=[Depends(require_auth)])
@app.post("/coordinator/checkpoint/{run_id}/approve", dependencies=[Depends(require_auth)])
@app.post("/coordinator/checkpoint/{run_id}/reject", dependencies=[Depends(require_auth)])
@app.get("/events", dependencies=[Depends(require_auth)])   # also reads ?token=
```

**`coordinatorApi.ts`** — pass the token in requests:

```ts
const TOKEN = import.meta.env.VITE_DASHBOARD_TOKEN || '';

// fetch → Authorization header: Bearer <token>
headers: { 'Content-Type': 'application/json', ...(TOKEN && { Authorization: `Bearer ${TOKEN}` }) }

// EventSource → query param
new EventSource(`${API_URL}/events${TOKEN ? `?token=${TOKEN}` : ''}`)
```

**`.env`**:
```
DASHBOARD_TOKEN=<secret generated with openssl rand -hex 32>
VITE_DASHBOARD_TOKEN=<same secret>
```

---

#### Option B — httpOnly Session cookie ← to be implemented for internet exposure / multi-user

**When to migrate**: as soon as there are multiple distinct users, different roles (read-only vs. checkpoint approval), or direct internet exposure.

**Recommended Stack**:
- `fastapi-users` or manual implementation with `itsdangerous.TimestampSigner`
- `httpOnly; Secure; SameSite=Strict` Cookie → automatically transmitted by `EventSource` and `fetch`
- No token in query param (safer)
- Svelte login page (`/login`) → `POST /auth/login` → set cookie → redirect `/`
- `POST /auth/logout` route → invalidates the cookie

**Session Schema**:

```
POST /auth/login   { username, password }
  → checks in a users table (SQLite or .htpasswd)
  → generates a signed token (JWT or itsdangerous signed cookie, 8h TTL)
  → Set-Cookie: session=<token>; HttpOnly; Secure; SameSite=Strict

GET /api/*  /events  /coordinator/*
  → middleware verifies the session cookie
  → 401 → redirect /login

POST /auth/logout
  → Set-Cookie: session=; Max-Age=0
```

**Role Management (optional)**:

| Role | Permissions |
|---|---|
| `viewer` | read `/api/status`, `/events` |
| `operator` | + send commands `/api/command` |
| `admin` | + approve/reject checkpoints |

**Points of attention**:
- Renew the cookie before expiration (sliding session or refresh token)
- HTTPS mandatory (`Secure` flag of the cookie)
- Rotation of the signing secret if compromised
- Rate-limiting on `POST /auth/login` (e.g., `slowapi`)
