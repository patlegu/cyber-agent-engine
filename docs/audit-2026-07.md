# Audit qualité & sécurité — cyber-agent-engine

Date : 2026-07-22
Portée : dépôt complet (`server.py`, `coordinator/`, `agents/`, `clients/`, `dashboard/`).
Méthode : audit statique par lecture (aucune exécution), preuves en `fichier:ligne`.

> **Nature du dépôt** : *architecture showcase* public (GitHub `patlegu/cyber-agent-engine`,
> 2★ / 1 fork). Ce rapport évalue l'état réel du code, pas seulement sa lisibilité.

---

## Synthèse

Le projet a une **substance réelle** (~55 fichiers Python, code fonctionnel) et une
**fracture de qualité nette** : la couche `clients/` est de facture référence (subprocess
durci, exceptions typées, validation `ipaddress`), tandis que les couches `coordinator/`,
`agents/` et `dashboard/` accumulent des trous de sécurité et de robustesse. Le composant
que le projet met en avant — le coordinateur — est **fonctionnellement impressionnant mais
pas digne de confiance en l'état** : il pilote de vrais équipements réseau via des points
d'entrée non authentifiés.

**Non exploitable en production sans une passe de sécurité.** En l'état, également **non
réutilisable légalement** (aucune licence).

## Findings Critiques (à traiter avant toute réexposition)

### C1 — `/api/command` du coordinateur n'a **aucune authentification**
`coordinator/server.py:565-566` (+ `/events` 600-601, `/api/logs` 633-634). Ces routes
n'ont pas le `dependencies=[Depends(verify_api_key)]` que portent toutes les autres, alors
que le docstring du module (`server.py:1-10`) affirme « toutes les routes sauf /health
exigent X-API-Key ». `/api/command` déclenche `pilot.react()` — **n'importe qui peut piloter
l'agent**, qui a un accès *écriture* réel à OPNsense/WireGuard/CrowdSec.

### C2 — Valeurs sensibles réelles loguées puis exposées sans auth
`clients/tool_agent_client.py:192-197` logue à INFO les entités NER **non anonymisées**
(`agents/ner_extractor.py:44-46` : « n'anonymise pas ») — communautés SNMP, users VPN,
hashes, IP, MAC. Le buffer de logs est servi par `/api/logs` (voir C1, sans auth). AnonyNER,
censé être la frontière de confidentialité, ne l'est pas. Un `GET` non authentifié suffit à
divulguer ces valeurs.

### C3 — Secret de build publié dans le dépôt *(corrigé le 2026-07-22)*
`dashboard/static/assets/index-3fb80e0f.js` embarquait un token 64-hex
(`VITE_DASHBOARD_TOKEN` inliné par Vite) commité dans l'historique public et présent dans le
fork. **Statut** : bundle désuivi + gitignoré (`9497e6e`). Le token n'était déployé nulle
part → rien à révoquer. Reste dans l'historique git et le fork (valeur morte).

### C4 — Auth fail-open + `/api/status` ouvert (dashboard)
`dashboard/app.py:25-26` : si `DASHBOARD_TOKEN` n'est pas défini, l'auth est **désactivée**
(déploiement par défaut = aucune authentification). `dashboard/app.py:63` : `/api/status`
n'a aucune dépendance d'auth et expose CPU/mémoire/disque + joignabilité OPNsense/vLLM.

## Findings High

- **Judge fail-open** (`coordinator/judge.py:152-160`) : `CAPValidator` renvoie
  `JudgeVerdict.ok()` quand son index de capacités est vide ou l'agent non indexé. Un accroc
  transitoire de récupération des capacités **désactive silencieusement toute validation** —
  précisément quand on la voudrait plus stricte.
- **Substitution de directive** (`coordinator/pilot.py:392-404`, `686-694`) : le chemin
  « reformulation structurée » appelle `execute_structured()` **sans** passer par le Judge,
  et prend `function` de la sortie LLM libre. Un humain approuve une fonction, le LLM peut en
  exécuter une autre. *Actuellement code mort* (voir M1) mais atteignable dès que `run()` est
  recâblé.
- **`verify_ssl=False` par défaut** : CrowdSec (`clients/crowdsec_client.py:30`,
  `agents/crowdsec_agent.py:117`) et l'appel OPNsense du dashboard (`dashboard/app.py:82`,
  `verify=False`). Tous les autres clients sont à `True` — incohérence à aligner.

---

## Findings Medium

- **M1 — Code mort** *(corrigé)* : `agents/opnsense_agent_reference.py` (1228 lignes, zéro
  importeur, sous-ensemble strict du package `agents/opnsense/`). Supprimé (`b4b3040`). Le
  coordinateur porte aussi ~350 lignes de chemin `plan()/run()` mort (`pilot.py`), non
  supprimées car elles hébergent la substitution de directive — à traiter avec le fix Judge.
- **M2 — Duplication mixins** : ~91 répétitions du bloc `try/except/simulation-fallback`
  dans `agents/opnsense/*.py` (~700 lignes). Le projet sait factoriser (`_decorators.py`,
  `safety_snapshot`) — extraire un décorateur.
- **M3 — Garde d'approbation asymétrique** (`pilot.py:60-73`) : seules les mutations
  `del_/disable_/block_` exigent une validation humaine. `add_nat_port_forward`,
  `add_wireguard_peer`, `enable_*` s'exécutent **seules** — l'IA peut *ouvrir* une exposition
  sans checkpoint, mais a besoin d'un humain pour la fermer.
- **M4 — Pas de validation d'entrée OPNsense** : ni type ni format avant les appels API
  (IP, params). CrowdSec, lui, valide via `ipaddress`.
- **M5 — Robustesse transport** (`tool_agent_client.py:117-153`) : retry sur
  `TimeoutException` seulement (pas ConnectError/RemoteProtocolError) ; pas de limite de
  taille de réponse (`resp.json()` désérialise tout avant validation) ; `CheckpointStore`
  (`state.py:150-163`) sans TTL/éviction → croissance mémoire non bornée.
- **M6 — Bundle compilé commité** *(corrigé via C3)* : `dashboard/static/` avait divergé de
  `frontend/src/` ; `templates/index.html` est le fallback (à conserver tant que le bundle
  n'est pas versionné).

---

## Findings Low

- **Zéro test** dans tout le dépôt (pas de `tests/`, `pytest.ini`, `conftest.py`). Pour un
  composant présenté comme « le cœur de l'architecture », avec une surface d'auth vive et un
  validateur fail-open, c'est le **levier le plus fort** : aucun des findings ci-dessus n'est
  gardé par une régression.
- Fonctions longues (`pilot.py` : `_reformulate` ~150 l., `_react_loop` ~110 l.,
  `execute_plan` ~103 l.).
- `except:` nus dans `dashboard/app.py:75,106` (attrapent `BaseException`, y compris
  `CancelledError`).
- Restes de migration : en-tête `requirements.txt` (« LORA FACTORY - Standalone… »),
  chemins `factory/agents/` dans du texte, incohérences FR/EN.
- **Aucune licence** — bloque toute réutilisation légale d'un dépôt public forké.

## Ce qui est bien fait (à préserver comme référence)

- `clients/crowdsec_client.py` : subprocess durci (liste, jamais `shell=True`, allowlist de
  sous-commandes, regex anti-traversal, timeout), validation IP `ipaddress`. Fichier
  exemplaire.
- `clients/opnsense_api_client.py` : couche requête à exceptions typées étroites.
- CAP v1 en modèle Pydantic (`coordinator/models.py`) ; réponses agent validées Pydantic.
- Flux d'approbation *race-safe* (`coordinator/server.py:406-412`).
- `_needs_approval()` ignore volontairement l'avis du LLM sur `requires_approval` — bonne
  posture défensive.
- Typage cohérent partout.

## Ordre de traitement recommandé

1. **Sécurité d'abord** (C1, C2, C4) : câbler `verify_api_key` sur les 3 routes du
   coordinateur ; retirer les entités des logs INFO (ou anonymiser réellement) ; fermer le
   fail-open de l'auth dashboard + protéger `/api/status`.
2. **Judge fail-closed** (High) + supprimer le chemin mort qui héberge la substitution de
   directive.
3. **Licence** : décision légale, un fichier — débloque la réutilisation.
4. **Aligner `verify_ssl=True`** partout.
5. **Amorcer des tests** : au minimum sur le coordinateur (auth, Judge, approbation) — c'est
   ce qui rend les fixes ci-dessus durables.
6. Hygiène : décorateur pour les mixins (M2), validation OPNsense (M4), bornes transport/état
   (M5), nettoyage des restes de migration.

**Déjà corrigé** : C3 (bundle désuivi, `9497e6e`), M1 (1228 lignes de code mort supprimées,
`b4b3040`).

## Suivi — sous-projet A (cœur de confiance) livré 2026-07-22

Le redesign de la chaîne de confiance (spec `docs/superpowers/specs/2026-07-22-coeur-confiance-surete-design.md`)
a été construit sous un nouveau package **`core/`** (branche `feat/trust-core`),
test-first, 10 tasks.

**Résultat : CQI `core/` = 9.2/10 (Excellent)**, cible >9 atteinte dès le départ.
49 tests, couverture 99 % (modules de sécurité — politique, exécution,
orchestrateur, tokens, auth — à 100 %), ruff propre, mypy strict propre.

**Livré** (`core/`) :
- `policy/` — moteur déterministe fail-closed (défaut deny), politique déclarative
  en données évaluée par une fonction pure ; catalogue de capacités + chargement
  qui refuse de démarrer sur politique invalide.
- `tokens/` — tokenisation réversible par session ; le LLM et les logs ne voient
  que des jetons ; détokenisation au seul point d'exécution.
- `auth/` — clé API temps-constant, fail-closed au démarrage, dépendance globale
  (test d'introspection : aucune route sans auth).
- `execution/` — `Authorized` infalsifiable par les types (sentinelle + `__slots__`) ;
  la frontière n'accepte qu'un verdict autorisé.
- `audit/` — puits append-only, jetons uniquement (test de propriété anti-fuite).
- `approval/` — approbation humaine liée au **hash canonique** de l'intention
  (anti-substitution de directive), `Approval` immuable, fail-closed.
- `orchestrator.py` — composition mince tokenize→propose→policy→approve/execute→audit.

**Non fait dans A (par conception)** : le `coordinator/` legacy (ReAct, Judge
fail-open, routes sans auth) n'est **pas encore débranché** — son remplacement par
le câblage sur `core/` dépend du contrat **CAP v2 (sous-projet B)**. Les 4 Critiques
et les High de l'audit d'origine restent donc présents dans le code legacy tant que
B n'a pas basculé le coordinateur sur `core/`.

**Dette consignée** (sous-projets suivants) :
- Rejeu double d'une approbation (reste `approved` après exécution) — à fermer
  (B) : marquer l'approbation consommée.
- Approbations jamais résolues : fuite de vault en mémoire, besoin d'un TTL/horloge
  (le store est volontairement pur/sans horloge) → store persistant/borné (D).
- `matched_rule.reason` perdu à l'audit post-approbation (traçabilité de la règle
  déclenchante) — Minor.
- Nettoyage legacy : `agents/`/`coordinator/`/`dashboard/` non lintés (1176 erreurs
  ruff), licence absente, `verify_ssl` fail-open — chantiers D et hygiène.
